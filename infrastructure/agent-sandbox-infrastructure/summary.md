# How We Built Secure, Scalable Agent Sandbox Infrastructure

**Author:** Larsen Cundric (@larsencc)
**Date:** Feb 27, 2026
**Source:** https://x.com/larsencc/status/2027225210412470668

## Key Takeaways

- There are two isolation patterns for code-executing agents: isolate just the dangerous tool behind an HTTP boundary, or isolate the entire agent in a sandbox with zero secrets — the latter is strictly superior because the agent becomes disposable and stateless.
- Unikraft micro-VMs boot in under one second, support scale-to-zero, and give each agent session a dedicated VM with only three env variables (session token, control plane URL, session ID) — no cloud keys or DB credentials ever enter the sandbox.
- The same container image runs in both production (Unikraft) and local dev/evals (Docker), toggled by a single config flag, which eliminates environment drift.
- A stateless FastAPI control plane acts as a credential-holding proxy: the sandbox cannot reach the outside world except through it, and every request is authenticated with a session-scoped bearer token.
- LLM conversation history lives in the control plane's database, not in the sandbox — so sandboxes can be killed and respawned without losing context, and cost caps are enforced server-side.
- File sync uses presigned S3 URLs generated by the control plane, allowing the sandbox to upload and download files without ever seeing AWS credentials.
- The extra network hop per operation is negligible compared to LLM latency, making the security-complexity tradeoff strongly favorable.

## One-Line Summary

Agents that execute arbitrary code should run in zero-secret, disposable sandboxes that talk to a credential-holding control plane, trading a trivial network hop for complete isolation.
