# Harness Engineering: Leveraging Codex in an Agent-First World

**Author:** OpenAI
**Date:** February 2026
**Source:** https://openai.com/index/harness-engineering/

---

The article describes what OpenAI learned during a five-month internal experiment in which a small team built and shipped a production beta product containing roughly **one million lines of code** -- with **zero manually written source code**. Every line (application logic, tests, CI configuration, documentation, observability, tooling) was generated by Codex agents. The post coins and defines the discipline of **"harness engineering"**: designing the systems, scaffolding, constraints, and feedback loops that allow AI coding agents to do productive, reliable work at scale.

## Key Statistics

- **~1,000,000 lines** of agent-generated code
- **~1,500 pull requests** opened and merged
- **3 engineers** initially on the team
- **~3.5 PRs per engineer per day** average throughput
- **5-month** timeline from start to beta

## The Engineer's Role Shifts

Human engineers no longer implement features directly. Their job becomes: prioritizing work, translating user feedback into acceptance criteria, validating outcomes, and -- when agents struggle -- identifying what is missing (tools, guardrails, or documentation) and feeding it back into the repository.

## Context Management Is the Central Challenge

One of the earliest lessons was to "give Codex a map, not a 1,000-page instruction manual." Context is a scarce resource. A giant AGENTS.md file crowds out the task, the code, and the relevant docs. When everything is marked "important," nothing is.

The solution: treat AGENTS.md as a short table of contents with pointers to a structured `docs/` directory (design docs, execution plans, product specs, references), enabling **progressive disclosure** where agents discover context as needed.

## Depth-First Development

The workflow requires breaking larger goals into small building blocks, having the agent construct those blocks, then using the results to unlock increasingly complex tasks in a compounding fashion.

## Architectural Constraints — Mechanical Enforcement

OpenAI enforces a strict layered architecture across business domains with validated dependency directions:

> **Types -> Config -> Repo -> Service -> Runtime -> UI**

Structural tests validate compliance and prevent violations. Dependencies flow in a controlled, one-directional sequence.

## Custom Linters and "Taste Invariants"

Rather than relying on documentation or style guides alone, the team enforces rules through custom linters and structural tests, plus a small set of "taste invariants":
- Structured logging enforcement
- Naming conventions for schemas and types
- File size limits
- Platform-specific reliability requirements

Because the lints are custom, their error messages are written specifically to **inject remediation instructions into the agent's context** -- effectively teaching the agent how to fix violations when they occur.

## Repository Optimized for Agent Legibility

Since the repository is entirely agent-generated, it is optimized first for Codex's legibility, not human readability. The goal is to make it possible for an agent to reason about the full business domain directly from the repository structure itself.

## CI as a Continuous Quality Feedback Loop

On a regular cadence, background Codex tasks scan for deviations, update quality grades, and open targeted refactoring PRs. Most of these can be reviewed in under a minute and auto-merged.

## Observability Wired into Agent Runtimes

The team wired Chrome DevTools Protocol into agent runtimes for DOM snapshots and screenshots, and provided local observability stacks with queryable APIs (LogQL, PromQL, TraceQL), giving agents the ability to inspect and debug their own output.

## Documentation as the System of Record

The knowledge base lives in a structured `docs/` directory that is treated as the single source of truth. Cross-linked design and architecture documentation is mechanically enforced with linters and CI validation.

## Key Takeaway

The competitive advantage ("moat") is no longer in the code itself but in the **control infrastructure** -- the constraints, documentation structures, linters, CI pipelines, and feedback loops that enable AI agents to produce reliable, architecturally coherent software at scale.
