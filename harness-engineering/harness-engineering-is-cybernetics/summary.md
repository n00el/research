# Harness Engineering Is Cybernetics

**Author:** George (@odysseus0z)
**Date:** March 7, 2026
**Source:** https://x.com/odysseus0z/status/2030416758138634583

## Key Takeaways

- "Harness engineering" (designing environments and feedback loops for AI agents instead of writing code directly) is the same cybernetic pattern as Watt's governor and Kubernetes controllers -- you stop turning the valve and start steering.
- LLMs are the first tool that can both sense and act at the architectural level of a codebase, closing the feedback loop where the important decisions are made.
- When agents produce bad output, the problem is almost never capability -- it's that domain knowledge (conventions, architecture, taste) is locked in engineers' heads and hasn't been externalized into machine-readable form.
- The fix is making judgment machine-readable: architecture docs, custom linters with remediation instructions, golden principles, and codified standards that the harness can enforce automatically.
- Engineering best practices (documentation, testing, codified architecture decisions) were always correct, but agentic engineering makes the penalty for skipping them unbearable -- agents amplify technical debt at machine speed.
- The engineer's role shifts from generating code to evaluating it: specifying what "correct" looks like, recognizing when output misses, and judging direction (generation-verification asymmetry).

## One-Line Summary

Harness engineering is just the latest instance of cybernetics -- the engineer's job shifts from writing code to designing the feedback loops and externalizing the judgment that agents need to produce correct output.
